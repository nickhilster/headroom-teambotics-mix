Who is the primary new user we're designing for?

The onboarding flow should be optimized for one persona first.

A

App Developer

Building an LLM-powered product (chatbot, RAG pipeline, agent). Arrives via PyPI or npm. Wants to drop Headroom into their existing stack with minimal friction. Success = compression in their app's code path.

Their first question

  • "How do I call this from my code?"
  • "Will it break my message format?"

Their blocker

  • Picking the right integration pattern
  • Understanding TS proxy requirement
B

AI Power User

Uses Claude Code, Codex, or Cursor heavily every day. Not building an app — wants cheaper, faster agent sessions. Arrives via README or word of mouth. Success = headroom wrap claude just works.

Their first question

  • "Will this slow down my agent?"
  • "How do I see my savings?"

Their blocker

  • Trust (will it mess with my sessions?)
  • Visibility into what it's doing
C

Both — separate paths

The onboarding should detect or ask which mode the user needs and branch early. App devs go to library/proxy docs; power users go to wrap/install flow.

Benefit

  • No one reads irrelevant content
  • Each path can be tight and focused

Cost

  • More content to maintain
  • Users may pick wrong path
